Custom tooling
automation without noise

Generic tools leave generic traces and do generic things. Custom tooling does exactly what you need and nothing else. The difference matters operationally.

why custom over generic

Every tool you run has a signature. Nmap has one. Metasploit has one. Even curl has default headers that fingerprint it. Custom tooling gives you control over that signature. It also means you understand every line of what you're executing.

Beyond stealth, custom tools do exactly one thing well. No unnecessary features, no bloat, no dependencies you don't control. Minimal surface, maximum efficiency.

the structure that scales

Every tool we build follows the same skeleton. Argument parsing, logging, error handling, clean output. Consistent structure means less cognitive overhead when reading code you wrote six months ago.

#!/usr/bin/env python3
# gh0stnet tool skeleton
 
import argparse
import sys
import logging
from datetime import datetime
 
logging.basicConfig(
format='%(asctime)s [%(levelname)s] %(message)s',
datefmt='%Y-%m-%d %H:%M:%S',
level=logging.INFO
)
log = logging.getLogger(__name__)
 
def parse_args():
p = argparse.ArgumentParser(description='tool description')
p.add_argument('-t', '--target', required=True)
p.add_argument('-o', '--output', default='report.txt')
p.add_argument('-v', '--verbose', action='store_true')
return p.parse_args()
 
def run(args):
log.info(f"target: {args.target}")
# core logic here
 
if __name__ == '__main__':
args = parse_args()
if args.verbose:
logging.getLogger().setLevel(logging.DEBUG)
try:
run(args)
except KeyboardInterrupt:
log.info("interrupted. exiting clean.")
sys.exit(0)

practical example — host monitor

A tool that watches a list of hosts, logs state changes and timestamps. Simple. Does one thing. No external dependencies beyond stdlib.

#!/usr/bin/env python3
# host state monitor — gh0stnet
 
import subprocess, time, json
from datetime import datetime
 
HOSTS = ["10.0.0.1", "10.0.0.2", "10.0.0.3"]
STATE = {h: None for h in HOSTS}
INTERVAL = 30 # seconds
 
def ping(host) -> bool:
r = subprocess.run(
['ping', '-c', '1', '-W', '2', host],
capture_output=True
)
return r.returncode == 0
 
def log_event(host, prev, curr):
ts = datetime.now().strftime('%Y-%m-%d %H:%M:%S')
status = "UP" if curr else "DOWN"
print(f"[{ts}] {host} → {status}")
with open('host_log.jsonl', 'a') as f:
f.write(json.dumps({"ts": ts, "host": host, "state": status}) + '\n')
 
while True:
for host in HOSTS:
curr = ping(host)
if curr != STATE[host]:
log_event(host, STATE[host], curr)
STATE[host] = curr
time.sleep(INTERVAL)
 
→ [2024-01-12 14:32:01] 10.0.0.1 → UP
→ [2024-01-12 14:34:31] 10.0.0.3 → DOWN
→ [2024-01-12 14:35:01] 10.0.0.3 → UP
JSONL output matters. One JSON object per line means you can pipe it, grep it, parse it with jq, feed it to anything. Structured output is more valuable than pretty output.

automation principles

  • One tool, one job — scope creep makes tools unpredictable
  • Explicit over implicit — no magic, no hidden behavior
  • Fail loudly — silent failures are worse than crashes
  • Log everything — if it ran, there should be a record
  • No hardcoded values — config files or arguments, never literals in logic
  • Reproducible — same input always produces same output

the last principle

Read every line of code you run. Every dependency you import. If you don't understand what a piece of code does, you don't control what it does.

Automation you don't understand is just someone else's code running on your machine.
