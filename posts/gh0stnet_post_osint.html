2024.02.14 osint recon

OSINT passive recon
building a methodology from zero

Most people treat OSINT as a toolbox. It's not. It's a discipline. Tools are the last step, not the first.

before you open a terminal

The biggest mistake in OSINT is jumping to tools before defining scope. What are you looking for? What's the objective? Unfocused recon generates noise, not intelligence.

Passive recon means leaving no footprint on the target. No direct requests to their infrastructure. No queries that show up in their logs. You're a ghost reading publicly available information — nothing more.

layer 01 — surface mapping

Start with what's already indexed. DNS records, WHOIS data, certificate transparency logs. These exist whether you look at them or not.

# full DNS enumeration
dig target.com ANY +noall +answer
dig target.com MX +short
dig target.com TXT +short
 
# WHOIS — registrar, dates, nameservers
whois target.com | grep -E "Registrar|Created|Expires|Name Server"
 
# certificate transparency — subdomain discovery
curl -s "https://crt.sh/?q=%.target.com&output=json" \
| python3 -c "import sys,json; \
[print(e['name_value']) for e in json.load(sys.stdin)]" \
| sort -u
 
→ mail.target.com
→ dev.target.com
→ api.target.com
→ staging.target.com
crt.sh is underrated. Certificate transparency logs are public by design. Every SSL cert ever issued for a domain is logged. Subdomains that were never meant to be found, found.

layer 02 — google dorks

Google has indexed things the target forgot existed. Old documents, exposed directories, cached pages from three years ago. Dorks are surgical queries, not brute force.

dork what it finds
site:target.com filetype:pdf Public documents — often contain metadata, author names, internal paths
site:target.com inurl:admin Admin panels indexed by Google. More common than it should be.
site:target.com intitle:"index of" Open directory listings. Files exposed without auth.
"@target.com" filetype:xls Spreadsheets with email lists. HR gold.
site:github.com "target.com" password Hardcoded credentials in public repos. Happens constantly.
cache:target.com Google's cached version. Pages that were deleted but not forgotten.

layer 03 — metadata extraction

Public documents are intelligence. PDFs, Word files, images — all carry metadata that most people never think about before publishing.

# download and analyze document metadata
wget -q https://target.com/document.pdf
exiftool document.pdf
 
Author : john.smith
Creator : Microsoft Word 2016
Producer : Windows 10 Enterprise
Create Date : 2023:11:14 09:23:41
Company : Target Corp Internal
 
# image metadata — GPS coordinates, device info
exiftool image.jpg | grep -E "GPS|Camera|Software|Date"

Username from a PDF author field → LinkedIn search → org chart. Windows version from metadata → patch level inference → vulnerability surface. One document. Multiple intelligence threads.

layer 04 — historical intelligence

The internet has memory. Targets don't always count on that.

  • Wayback Machine — pages deleted years ago, still cached
  • Google cache — recent deletions often still indexed
  • Shodan — historical port scan data, even if firewall rules changed
  • GitHub — deleted repos sometimes recoverable, commits are forever
  • Pastebin archives — credentials and keys dumped and forgotten
# check Wayback Machine availability
curl -s "http://archive.org/wayback/available?url=target.com" \
| python3 -c "import sys,json; \
d=json.load(sys.stdin); \
print(d['archived_snapshots']['closest']['url'])"

the output that matters

Raw data is not intelligence. Recon ends with a structured report:

  • Asset inventory — domains, subdomains, IPs, services
  • Personnel data — names, usernames, email patterns
  • Technology stack — inferred from headers, metadata, job postings
  • Exposure summary — what's public that probably shouldn't be
  • Attack surface — ranked by exploitability, not just existence

If you can't turn your recon into a structured brief, you were collecting data, not doing intelligence. Know the difference.
